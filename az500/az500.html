Dynamic Membership Rules

What: Users/devices can be automatically added to a group based on attribute-based rules (e.g., department = "Engineering", jobTitle = "Developer").

How: Rules are configured the same way for both Security Groups and M365 Groups. When a user attribute matches the rule, they are automatically added/removed.

License needed: Entra ID P1 or P2

Client Secrets vs. Certificates

Client Secrets
  • Password-like string for app authentication
  • Simpler to set up
  • Less secure — string can be shared/leaked
  • Have expiry date (max 24 months)
Certificates
  • Public/private key pair
  • More secure — private key never shared
  • Preferred for production environments
  • Certificate stored in app; private key in app's secure store

Managed Identities

Problem solved: Traditionally, apps needing to access Azure resources (like Key Vault) needed credentials embedded in code — a security risk.

Managed Identity: Azure manages the identity automatically. No credentials in code. The resource gets an SP in Entra ID, and you assign RBAC to that SP.

System-Assigned
  • Tied to a specific Azure resource
  • Deleted when resource is deleted
  • 1:1 relationship (one identity per resource)
  • Enabled directly on the resource
User-Assigned
  • Standalone Azure resource
  • Can be assigned to multiple resources
  • Lifecycle independent of assigned resources
  • Created separately, then attached
⭐
How it works

Resource (e.g., VM) → Managed Identity created in Entra ID → Assign RBAC on target resource (e.g., Key Vault) to that managed identity → App code uses Azure SDK to get token automatically — no credentials in code.

Multi-Factor Authentication (MFA)

MFA adds an extra layer of authentication beyond username/password. Even if credentials are compromised, attacker still cannot authenticate without the second factor.

Security Defaults: Free tier feature that enables baseline MFA for all users. Replaces the old baseline Conditional Access policies.

Conditional Access

Conditional Access is if-then policies: IF a user/condition is detected, THEN enforce a control. Think of it as the zero-trust policy engine for access decisions.

ComponentExamples
Signals (IF)User, Group, IP/Location, Device platform, Application, Risk level
Controls (THEN)Require MFA, Block access, Require compliant device, Require Hybrid Azure AD join
💡
Important Note

All authentication-related settings are now managed under Authentication Methods Policies section in Entra ID. Named Locations can be defined (trusted IPs, countries) and used as signals in Conditional Access policies.

⭐
Exam: Requires P1/P2

Conditional Access requires Entra ID P1 or P2. Security Defaults (free) are mutually exclusive — you must disable Security Defaults to use Conditional Access.

Identity Protection

Detects inconsistencies and anomalies in the sign-in process using machine learning. Generates risk signals that can be used in Conditional Access policies.

User Risk
  • Probability that an identity is compromised
  • Examples: leaked credentials, unusual behavior
  • Policy action: Require password change
Sign-In Risk
  • Probability that a sign-in is not by owner
  • Examples: atypical travel, unfamiliar location, anonymous IP
  • Policy action: Require MFA or Block
⭐
Requires Entra P2

Identity Protection is a P2-only feature. Risky users and risky sign-ins dashboards are available to review and remediate flagged identities.

Network Security

Azure Virtual Network (VNet)

An isolated network on the Azure cloud. Resources hosted in a VNet communicate privately. Scoped to a specific region and subscription.

🌐 VNet (Region + Subscription)
📦 Subnet 1 (e.g., WebSubnet — 10.0.1.0/24)
🖥 VM1, VM2
📦 Subnet 2 (e.g., AppSubnet — 10.0.2.0/24)
🖥 VM3, App Service
📦 GatewaySubnet (reserved for VPN/ExpressRoute Gateways)

Network Security Groups (NSGs)

NSGs filter network traffic between Azure resources in a VNet. They contain Inbound and Outbound security rules.

NSG Rule Properties

PropertyDescription
NameUnique name for the rule
Priority100–4096. Lower number = Higher priority
Source/DestinationIP, CIDR range, Service Tag, or ASG
ProtocolTCP, UDP, ICMP, or Any
Port RangeSingle port, range, or wildcard (*)
ActionAllow or Deny
🔴
Critical NSG Rules — Must Know
  • Rules are evaluated in priority order — ONCE A RULE MATCHES, PROCESSING STOPS
  • Lower priority number = Higher priority (100 beats 200)
  • NSG can be associated with a Subnet OR a NIC — NOT the entire VNet
  • If multiple NSGs exist, rules must be in ALL relevant NSGs to allow traffic

Default NSG Rules (Cannot Be Deleted)

PriorityNameInbound/OutboundAction
65000AllowVNetInBoundInboundAllow
65001AllowAzureLoadBalancerInBoundInboundAllow
65500DenyAllInBoundInboundDeny
65000AllowVNetOutBoundOutboundAllow
65001AllowInternetOutBoundOutboundAllow
65500DenyAllOutBoundOutboundDeny

Application Security Groups (ASGs)

ASGs are an extension to NSGs that allow you to group VMs together logically and use those groups as source/destination in NSG rules. Instead of managing IP addresses, you manage app-layer groups.

Without ASGs
  • NSG rule: Allow 10.0.1.4, 10.0.1.5, 10.0.1.6 on port 80
  • Must update IPs manually as VMs change
  • Error-prone and complex
With ASGs
  • Group VMs into "WebServers" ASG
  • NSG rule: Allow "WebServers" ASG on port 80
  • VMs added/removed from ASG — rule stays same

VNet Peering

Connects two VNets together so resources can communicate directly using Microsoft backbone network (low latency, high bandwidth). Works same region and cross-region.

🔴
No Overlapping CIDR Blocks

When peering VNets, they CANNOT have overlapping IP address ranges (CIDR blocks). This is a common exam scenario — peering fails if address spaces overlap.

VPN Connections

VPN TypeUse CaseKey Details
Point-to-Site (P2S) Individual client computers → Azure VNet Protocols: OpenVPN, SSTP, IKEv2. Auth: Certificate, Entra ID, RADIUS+AD DS. Charged per hour.
Site-to-Site (S2S) On-premises network → Azure VNet (IPSec/IKE tunnel) Requires VPN device with public IP. Local Network Gateway needed. No overlapping subnets. Traffic encrypted via IPSec.

→

Site-to-Site: Key Requirements

  • On-prem VPN device with a publicly routable IP address (hardware like Cisco router, or software like Windows Server RRAS)
  • Local Network Gateway in Azure — provides routing info to VPN Gateway for on-prem network
  • No overlapping subnets between on-prem and Azure VNet
  • Connection uses IPSec tunnel for encryption
  • Different SKUs available for VPN Gateway — each has different throughput and pricing

ExpressRoute

Extends on-premises network to Microsoft cloud via a private, dedicated connection — NOT over the internet. Provided through a connectivity partner.

  • 2 connections per circuit for built-in redundancy
  • Azure Private Peering — access VMs via private IPs
  • Microsoft Peering — access public services like M365, Azure PaaS
  • Higher reliability, lower latency, consistent bandwidth compared to VPN over internet
⭐
ExpressRoute vs VPN — Key Difference

ExpressRoute = Private connection, NOT over internet. VPN = Encrypted tunnel OVER internet. ExpressRoute is for enterprises needing reliability and compliance.

Service Endpoints vs Private Endpoints

🔴
Critical — Heavily Tested

This comparison is a very common exam scenario. Know the differences deeply.

Service Endpoints
  • Traffic travels over Microsoft backbone network (not public internet)
  • Service remains a public service with public IP
  • Storage firewall used to restrict access to specific subnet
  • Does NOT block internet access to the service — still publicly reachable by others
  • No additional cost
  • Simpler to configure
Private Endpoints
  • Creates a private NIC in your VNet for the PaaS service
  • Traffic NEVER leaves your VNet — not even to MS backbone
  • Service gets a private IP address in your VNet
  • Effectively brings the service INTO your network
  • Charged per hour
  • Private DNS zone auto-created to map service FQDN to private IP
⭐
Setup Steps — Service Endpoint

1. Enable service endpoint on subnet (e.g., Microsoft.Storage). 2. On storage account firewall, allow access only from that subnet. Two-step process.

  • Azure CDN: Content caching at edge locations globally (simpler than Front Door).

Firewall & Network Architecture

Azure Firewall

A fully managed, cloud-native, stateful network security appliance. Unlike NSGs (which filter within VNets), Azure Firewall provides full network/application-level inspection and can look into requests from external users or internal users trying to reach the internet.

Azure Firewall Rule Types

1
DNAT Rules — Destination Network Address Translation. Used for port forwarding (like a bastion). Processed FIRST.
2
Network Rules — Layer 3/4 rules (IP, port, protocol). Processed SECOND.
3
Application Rules — Layer 7 (FQDNs, HTTP/HTTPS). Processed THIRD/LAST.
🔴
Firewall Rule Processing Order — Critical

Rule Collections Groups → processed by priority. Within a group, Rule Collections → processed by priority. Order is always: DNAT → Network → Application.

Routing Traffic Through Firewall

Creating a firewall alone doesn't route traffic through it. You must:

  1. Create the Azure Firewall in a hub VNet (needs AzureFirewallSubnet)
  2. Create a Route Table with a route: 0.0.0.0/0 → Azure Firewall private IP
  3. Associate the Route Table with the spoke subnets (e.g., WebSubnet)
  4. Create Firewall application/network rules to allow needed traffic
💡
Traffic destination note

Any traffic destined for the internet has the next hop IP 0.0.0.0/0. Adding a UDR (User Defined Route) for 0.0.0.0/0 pointing to the firewall private IP forces all internet-bound traffic through the firewall.

Azure Bastion

Fully managed service providing secure RDP/SSH connections to VMs from the Azure Portal — no public IP needed on the VM. Eliminates exposure of RDP/SSH ports to the internet.

  • Deployed in its own subnet: AzureBastionSubnet
  • Connections made via browser (HTML5) — no VPN client needed
  • Protects against port scanning as VM has no public IP
  • Supports both Windows (RDP) and Linux (SSH)

Hub-Spoke Network Architecture

Hub contains shared services: Azure Firewall, VPN/ExpressRoute Gateway, Azure Bastion, DNS. Spokes contain workloads and connect to Hub via VNet Peering. All traffic between spokes or to internet is routed through Hub Firewall via UDRs.

Azure SQL Managed Instance Networking

Azure SQL Managed Instance has native virtual network implementation. It is deployed inside a VNet, providing full VNet integration unlike Azure SQL Database (which is a public service by default).

  • Deployed in a dedicated subnet in a VNet
  • Near 100% compatibility with SQL Server Enterprise Edition
  • Supports private IP communication natively
  • Good for lift-and-shift migrations of on-prem SQL Server

Key Management & Encryption

Azure Key Vault

Managed service for storing secrets (connection strings, passwords), encryption keys, and certificates (HTTPS). Used to keep sensitive data out of code and config files.

Key Vault Access Models

Access Policies (Legacy)
  • Set permissions per object type (keys, secrets, certs)
  • Less granular — grant access to all keys or all secrets
  • Still supported but being replaced by RBAC
RBAC (Recommended)
  • Use Azure RBAC roles: Key Vault Officer, Key Vault Reader, Key Vault Secrets User, etc.
  • More granular — can scope to individual secrets
  • Centralized in IAM blade
  • Preferred for new deployments

Soft Delete & Purge Protection

FeatureDescription
Soft DeleteAllows recovery of deleted vaults/objects within a retention period (7–90 days configurable). Object goes to "deleted" state, not permanently removed.
Purge ProtectionWhen enabled, you CANNOT permanently delete a vault or object until the retention period expires. Prevents even admins from destroying data. Protects against ransomware/insider threats.
⭐
Deletion Process with Soft Delete

Step 1: Delete the vault → goes to soft-deleted state. Step 2: Purge the vault → permanently deleted. With Purge Protection enabled, Step 2 cannot happen until retention period expires.

Backup & Restore

Secrets, keys, and certificates can be backed up to an encrypted backup file and restored at any time. The backup is encrypted and can only be restored to the same Azure geography.

Key Rotation

Azure Key Vault supports automatic key rotation. You can configure a rotation policy to automatically rotate keys on a schedule. Integrates with services like Azure Storage for auto-rotating encryption keys.

Key Vault Network Security

  • By default Key Vault is accessible over the internet
  • Configure firewall rules to restrict to specific IP ranges or VNet service endpoints
  • Use Private Endpoint for fully private access (traffic stays in VNet)

Encryption Types & Concepts

Data States

Encryption at Rest
  • Protects data stored on physical disks
  • TDE for SQL databases
  • Azure Storage Service Encryption (SSE) — enabled by default
  • Azure Disk Encryption for VM disks
Encryption in Transit
  • Protects data moving over network
  • TLS/SSL for web apps and storage
  • Enforce HTTPS-only on storage and App Service
  • ExpressRoute (private) vs VPN (IPSec)

Azure Disk Encryption

Encrypts Windows VM disks using BitLocker and Linux VM disks using DM-Crypt. Keys stored in Azure Key Vault. Protects data if physical disks are removed from datacenter.

Customer-Managed Keys (CMK)

By default, Azure uses Microsoft-managed keys for encryption. With CMK, you bring your own keys stored in Azure Key Vault. You control the key lifecycle (rotation, revocation). If you revoke the key, data becomes inaccessible — giving you full control.

⭐
Platform-managed vs Customer-managed

Platform-managed keys: Azure handles everything. Customer-managed keys: You store keys in Key Vault, you control rotation/revocation. CMK is required for some compliance frameworks (HIPAA, PCI-DSS).

When a lock is placed on a Blueprint assignment, nobody can delete the resources — not even subscription owners. To delete resources, you must first unassign the blueprint, then delete the resources.

  • Policy Assignments — enforce compliance policies

CanNotDelete Lock
  • Users can read and modify the resource
  • Cannot delete the resource
  • Use when you want to prevent accidental deletion
ReadOnly Lock
  • Users can only read the resource
  • Cannot modify or delete
  • Similar to applying Reader RBAC to everyone
  • Most restrictive lock type

Microsoft Defender for Cloud

Overview

Cloud Security Posture Management (CSPM) + Cloud Workload Protection Platform (CWPP). Protects workloads across Azure, AWS, GCP, and on-premises.

  • Free 30-day trial available for enhanced plans
  • Free tier: Foundational CSPM, recommendations, secure score, regulatory compliance (Microsoft Cloud Security Benchmark)
  • Centralized Azure Policy based on Microsoft Cloud Security Benchmark (MCSB)

Secure Score

Numeric score (0–100%) representing your overall security posture. Based on security controls — each control has a max score. Remediating recommendations increases your score. Higher score = better security posture.

Just-In-Time (JIT) VM Access

Part of Defender for Servers. Locks down inbound ports (RDP 3389, SSH 22) on a VM by default. When access is needed, a request is made and access is granted for a limited time, then automatically revoked. Reduces attack surface.

⭐
JIT Workflow

Default: RDP/SSH ports BLOCKED by NSG rule. User requests access via portal/API → JIT creates temporary NSG rule allowing access from specific IP → Timer expires → Rule removed, ports blocked again.

Adaptive Application Controls

ML-powered feature in Defender for Servers. Analyzes what applications run on your VMs and creates an allowlist of known-good applications. Alerts when unexpected applications run (potential malware or unauthorized software).

Workflow Automation & Security Recommendations

Workflow Automation: Trigger Logic Apps in response to Defender for Cloud alerts or recommendation state changes. Used to auto-remediate or notify teams.

Security Recommendations: Actionable suggestions to improve security posture. Each recommendation shows affected resources, severity, and remediation steps.

Database & Storage Security

Azure SQL Database Security

Microsoft Entra Authentication for Databases

Instead of SQL logins, use Entra ID users/groups for database access. Set an Entra Admin for the SQL Server. Users can then connect using their Entra credentials (SSO, MFA, conditional access apply).

Database Auditing

Tracks database events and writes an audit log. Used for regulatory compliance and security investigation.

  • Can be set at database level or server level (server level applies to all DBs)
  • Audit logs sent to: Azure Storage Account, Log Analytics Workspace, or Event Hub
  • Important: When using Storage Account via portal, database server and storage account must be in the same region

Dynamic Data Masking

Hides sensitive data (e.g., credit cards, emails) from non-privileged users at query time. Data remains stored unmasked in the database. Only the query results are masked for non-privileged users.

⭐
Key Point

Dynamic Data Masking does NOT encrypt data — it just masks it in query output. Privileged users see real data. Available for Azure SQL Database.

Transparent Data Encryption (TDE)

Encrypts database files, backups, and transaction logs at rest on physical discs. Encryption/decryption happens in real-time, transparent to the application. Already enabled by default on Azure SQL Database.

Always Encrypted

Protects sensitive column data (credit cards, SSNs) even from DBAs. Data is encrypted client-side before being sent to the database. The database engine never sees plaintext.

Deterministic Encryption
  • Same plaintext → always same ciphertext

💡
Always Encrypted Setup

Must use SQL Server Management Studio (SSMS) to configure Always Encrypted — not available via Azure portal. The Always Encrypted Wizard guides through key creation and column selection.

Database Firewall Rules

Azure SQL Database has server-level and database-level firewall rules. By default, all external access is blocked. You must explicitly allow client IP ranges or Azure services.

Storage Account Security

Shared Access Signatures (SAS)

A SAS is a signed URI that grants time-limited, granular access to storage resources without sharing account keys. Fine-grained permissions (read, write, delete, list) on specific resources.

SAS LevelScopeUse Case
Account SASEntire storage account (blobs, files, queues, tables)Admin access to all services

Blob Security Features

FeatureDescription
Blob SnapshotsRead-only, point-in-time copy of a blob. Manual creation.
Blob VersioningAutomatically maintains a copy (immutable snapshot) on every write. Full history.
Soft Delete (Blobs)Protects individual blob/snapshot/version from accidental delete by retaining data for a specified period.
⭐
Snapshots vs Versioning vs Soft Delete

Snapshots = manual point-in-time copy. Versioning = automatic copy on every write. Soft Delete = recovery window after deletion. These can all be used together for comprehensive data protection.

Container Security

Container Fundamentals

Containers package application code + dependencies into a portable, isolated unit. Lighter than VMs (share OS kernel). Docker is the standard container runtime.

Azure Container Registry (ACR)

A private container registry (like a private Docker Hub) for storing Docker container images. Supports geo-replication for multi-region deployments.

ACR RBAC Roles

RolePrimary PurposePermissions
AcrPullConsumptiondocker pull only. For production apps, AKS clusters.
AcrPushDeploymentdocker push AND pull. For CI/CD pipelines.
ReaderView-OnlySee registry settings in portal. Cannot pull images.
ContributorManagementCan delete registry, change pricing tiers. No Data Plane access by default in ABAC-enabled registries.
🔴
ACR Role Trick — Exam Trap

The Reader role does NOT allow pulling images — it's view-only in portal. To pull images you need AcrPull. The Contributor role has no Data Plane access in newer registries — cannot pull/push images.

ACR Authentication Methods

  • Entra ID / RBAC — Recommended. Use managed identities for AKS/App Service
  • Admin account — Legacy. Single username/password. Not recommended for production.
  • Service Principal — For CI/CD pipelines and automation
  • Repository-scoped tokens — Fine-grained access to specific repositories

Azure Container Instances (ACI)

Run containers directly in Azure without managing VMs or orchestrators. Serverless containers. Good for simple tasks, batch jobs, and burst workloads. No cluster management needed.

Azure Kubernetes Service (AKS)

Managed Kubernetes service. Azure manages the control plane. You manage worker nodes (or use nodepools). Supports auto-scaling, rolling deployments, and advanced networking.

AKS Security Best Practices

  • Use Managed Identity for AKS to pull from ACR (AcrPull role)
  • Use Azure Policy for AKS to enforce pod security standards
  • Use Private Clusters — control plane API server only reachable via private IP
  • Enable Microsoft Defender for Containers for threat detection
  • Use Azure CNI for pod-level NSG enforcement

Microsoft Defender for Containers

Container Image Scanning
  • Scans images in ACR for known vulnerabilities (CVEs)
  • Scans on push and periodically
  • Integrates with Defender for Cloud recommendations
  • Registry-level scanning
Runtime Threat Protection
  • Detects suspicious activity in running containers
  • Crypto mining, privilege escalation, reverse shells
  • Works in AKS, Arc-enabled Kubernetes, EKS, GKE
  • Real-time alerts

🔴 Critical Rules — Never Forget

  • NSG priority: Lower number = Higher priority. 100 beats 4096. Once a rule matches, processing STOPS.
  • NSG association: Can ONLY attach to Subnet or NIC — NOT to an entire VNet.
  • VNet Peering: No overlapping CIDR blocks allowed.
  • Service Endpoint: Does NOT block internet traffic to the service — use Private Endpoint for that.
  • Private Endpoint: Creates NIC in your VNet. Traffic NEVER leaves your VNet. Chargeable per hour.
  • PIM max activation: 8 hours maximum per role activation.
  • M365 Groups: Cannot be added to a Security Group.
  • Blueprint locks: Even subscription owners cannot delete locked resources — must unassign blueprint first.
  • TDE: Already enabled by default on Azure SQL Database.
  • Dynamic Data Masking: Does NOT encrypt — masks at query result level only.
  • Always Encrypted: Must use SSMS to configure — not Azure portal.
  • Firewall rule order: DNAT → Network → Application rules (always in this order).
  • Azure Bastion subnet: Must be named exactly AzureBastionSubnet.
  • Synced users: Properties of on-prem synced users CANNOT be changed in Entra ID.
  • AcrPull/AcrPush: Reader role ≠ image pull. Contributor ≠ data plane access.
  • Security Defaults vs Conditional Access: Mutually exclusive — disable one to use the other.
  • Purge Protection: When enabled, even admins cannot permanently delete until retention period expires.
  • Soft Delete retention: 7–90 days (configurable).
